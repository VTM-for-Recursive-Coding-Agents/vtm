# VTM Structure
VTM structure helps link and define how types are used together.
## Purpose of  VTM Structure
These structures are designed to be a set of rules to decide when information needs to be verified, what is depended in each artifact, and how files are linked together.

## Core Concepts
Memory 
- Memory is the root unit for a VTM.
- VTM is centered on Memory records. Everything else exists to define, verify, or update a memory.

Types of Memory Intent
- each memory has a purpose catagory (artifact, claim, procedure) this tells the system how to interact, interpret, and verify the information.

Evidence Driven Verification
- Evidence-driven verification  is not a boolean flag by itself. It is evidence-backed, and results are captured as structured verification outputs tied to a memory.

Code Anchoring for Traceability
- CodeAnchor connects memories to concrete source locations and uses both structural and contextual hashes so VTM can detect drift over time.

Artifact Provenance and Dependencies
- ArtifactRef captures tool metadata, environment fingerprinting, and dependency links so verification can be reproducible and dependency-aware.

Lifecycle status transitions
- Memories move through a status lifecycle (Unverified, Valid, Invalid, Stale). Stale is a first-class concept for when upstream code or dependencies change.

Separation of model vs runtime behavior
- The type layer defines the contract, while the structure layer should define how components interact: memory management, anchor extraction, verifier execution, and result updates.

## System Components
Memory Store:
- creates and updates Memory records and evidence history.
Anchor Extractor: 
- parses source files and emits CodeAnchor records with structural and context hashes.
Artifact Resolver: 
- resolves ArtifactRef metadata and dependency links.
Verifier Runner:
- executes verifier strategy for a given memory.
Result Updater:
- writes VerificationResult and transitions memory status.

## End-to-End Flow
1) Create a memory with unverified status.
2) Attatch artifact references and codeAnchors references as evidence inputs.
3) Run Verifier Runner for the target memory. The (bash) script for verifiying is generated by the LLMs.
4) Produce verification results with status and evidence_results.
5) Update memory status and append evidence.
6) Mark related memories as stale when dependencies or anchors drift/change.

## Status Lifecylce
UNVERIFIED: new or not yet evaluated.
VALID: evidence currently supports the memory.
INVALID: evidence currently fails the memory.
STALE: prior result no longer trusted due to dependency or source drift.
Define allowed transitions:
1) UNVERIFIED to VALID
- When a memory is verified using the verifier runner
2) UNVERIFIED to INVALID
- When a memory fails to verify using the verifier runner
3) VALID to STALE
- When a dependency changes 
4) INVALID to STALE
- When a dependency changed 
5) STALE to VALID
- When a memory is verified using the verifier runner
6) STALE to INVALID
- When a memory fails to verify using the verifier runner

## Dependency and Staleness Model
Dependencies are upstream artifact IDs and anchor IDs.
- Any dependency hash or version change can trigger STALE.
- hash_ast drives semantic drift detection.
- hash_ctx drives contextual drift detection when relocation occurs.
